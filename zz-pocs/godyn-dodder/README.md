# godyn-dodder — dodder cross-package inlining density

A measurement, not a builder. Answers the open question left by `godyn-poc`'s
M5/R6 finding: *per-package CA isolation is bounded by Go's cross-package
inlining — so how much does dodder actually inline across package boundaries?*
That sets the realistic incremental-cache-reuse ceiling for a production
per-package builder targeting dodder, and whether the `-gcflags=all=-l`
(inlining-off) lever is worth offering.

The dodder source snapshot + vendor tree are **gitignored** (regenerated by
`just measure`); only this README + the recipe are tracked, to keep igloo lean.

## Method

`just measure` snapshots `dodder/go`, materialises its vendor tree from the nix
build (`cp -rL` of `nix build .#dodder.passthru.vendorEnv`), overlays the merged
go.mod, and runs `go build -gcflags=-m ./...` with the same env the gomod2nix
hook uses (`GO_NO_VENDOR_CHECKS=1 GOFLAGS=-mod=vendor GOPROXY=off`). The cgo
package `lib/bravo/pivy` fails to compile under `CGO_ENABLED=0` (it is compiled
last), so the inline data for everything else is complete.

## Findings (go 1.26.3, dodder @ working tree 2026-06)

- **6,753 inlinable functions; 14,236 inlined call sites** across dodder's build.
- Inlining is **dominated by dodder's own foundational generic packages**, not
  stdlib and not high-layer code:

  | inline source | call sites | kind |
  |---|---|---|
  | `collections_slice` | 4,291 | dodder generic container (**30% of all inlining**) |
  | `collections_value` / `collections_ptr` / `Slice[...]` / `Seq` | ~1,100 | dodder generic containers |
  | `document`, `catgut`, `ids`, `markl`, `doddish`, `cst`, `ui`, `genres`, `tai`, … | ~1,800 | dodder leaf/utility packages |
  | `errors`, `slices`, `strings`, `fmt`, `bytes`, `atomic`, `sync`, `os`, `bufio` | ~4,000 (~28%) | stdlib (stable — irrelevant to dodder cache churn) |

## Interpretation

Inlining density is **high but concentrated in low-churn foundational
packages** (generic containers + string/id/hash utilities), which are inlined
into nearly every consumer. The cache-isolation consequence is *churn-locality
dependent*:

- A change to a **foundational** package (e.g. `collections_slice`) re-flows
  inlined bodies into ~all consumers → poor incremental reuse **for that
  change**. But these packages rarely change.
- A change to a **high-layer** package (the upper NATO layers, where most dev
  iteration happens) that does not export heavily-inlined helpers stays well
  isolated → good incremental reuse.

So the per-package CA cache win is **real for typical high-layer edits, bounded
for foundational edits**.

## Implication for the production builder

- Offer a **`-gcflags=all=-l` (cross-package inlining off) build variant** for
  maximal incremental reuse — most valuable for CI / test-loop builds on a
  heavily-generic codebase like dodder, accepting a runtime-perf cost on the
  generic hot paths. Keep an inlined default variant (faster runtime,
  foundational-change-couples).
- Recommend benchmarking the `-l` runtime cost on a dodder hot path before
  choosing a default.
- This composes with the M5/R6 verdict: the iface split does **not** help here
  (Go's export data carries the inlinable bodies); `-l` is the only full-
  isolation lever.
